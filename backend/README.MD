# Mini Feed API

API REST che alimenta un'app React Native con feed paginata, like idempotenti e commenti.

---

## Avvio rapido

### Locale (senza Docker)

```bash
npm install
npm run dev            # http://localhost:3000
```

Nessun `.env` necessario in sviluppo: tutte le variabili hanno un default. Per personalizzare: `cp .env.example .env`.

> Redis ├и opzionale in locale. Se non ├и in esecuzione, la cache fallisce silenziosamente e la API funziona lo stesso senza di lei.

### Docker (API + Redis)

```bash
docker compose up --build        # build + avvio (log in console)
docker compose up --build -d     # in background
docker compose logs -f           # log in tempo reale
docker compose down              # arresto
```

Il compose avvia due servizi: `api` (Node.js) e `redis` (Redis 7 Alpine). La API aspetta che Redis superi il suo healthcheck prima di partire (`depends_on: condition: service_healthy`).

### Variabili d'ambiente

| Variabile | Obbligatoria | Default | Descrizione |
|-----------|--------------|---------|-------------|
| `JWT_SECRET` | Solo in produzione | default di sviluppo | Minimo 16 caratteri |
| `JWT_EXPIRES_IN` | No | `7d` | Scadenza del token |
| `PORT` | No | `3000` | Porta del server |
| `NODE_ENV` | No | `development` | `development` / `production` / `test` |
| `REDIS_URL` | No | `redis://localhost:6379` | URL di connessione a Redis |

### POST `/v1/auth/login`

Login mock via email. Ritorna un **JWT** firmato con HS256, valido 7 giorni.

La **password ├и opzionale** (login mock come da spec): senza password si entra solo con l'email; se viene inviata, deve essere corretta.

**Body**
```json
{ "email": "ada@ivory.test" }
```

**Risposta 200**
```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": { "id": "user-1", "email": "ada@ivory.test", "name": "Ada Lovelace", "role": "admin" }
}
```

Il payload del token contiene `{ sub: userId, role, iat, exp }`. Il middleware verifica la firma senza consultare nessuno store тАФ stateless.

```bash
curl -X POST http://localhost:3000/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "ada@ivory.test"}'
```

Utenti seed disponibili:

| Email | Password | Ruolo |
|-------|----------|-------|
| `ada@ivory.test` | `ivory1234` | `admin` |
| `alan@ivory.test` | `ivory1234` | `user` |

### GET `/v1/feed`

Feed paginata con cursor, ordinata dalla pi├╣ recente alla pi├╣ vecchia.

La prima pagina senza autenticazione viene servita dalla **cache Redis** (TTL 60s). Si invalida automaticamente a ogni like, unlike, commento o modifica.

**Query params**

| Param | Tipo | Default | Descrizione |
|-------|------|---------|-------------|
| `limit` | number | 10 | Massimo 20 |
| `cursor` | string | тАФ | ID dell'ultimo post ricevuto |

**Risposta 200**
```json
{
  "posts": [
    {
      "id": "post-10",
      "authorId": "user-2",
      "body": "Primer d├нa en el nuevo proyecto...",
      "createdAt": "2026-06-10T08:45:00.000Z",
      "likesCount": 62,
      "commentsCount": 0,
      "author": { "id": "user-2", "email": "alan@ivory.test", "name": "Alan Turing", "role": "user" },
      "likedByMe": false
    }
  ],
  "nextCursor": "post-9"
}
```

`nextCursor` ├и `null` quando non ci sono altre pagine. Il campo `author` non espone mai la password.

```bash
# Prima pagina (in cache su Redis)
curl "http://localhost:3000/v1/feed?limit=3"

# Pagina successiva
curl "http://localhost:3000/v1/feed?limit=3&cursor=post-8"
```

## Middleware

| Middleware | Descrizione |
|------------|-------------|
| `requestId` | Aggiunge un `X-Request-Id` univoco (UUID) a ogni request e response |
| `helmet` | Header di sicurezza HTTP |
| `cors` | Whitelist di origini (`localhost:8081`, `localhost:3000`) |
| `compression` | Compressione gzip delle response |
| `hpp` | Protezione contro HTTP Parameter Pollution |
| `morgan` | Log delle request HTTP, instradato attraverso Winston |
| `globalLimiter` | Rate limit globale: 100 request / 15min |
| `authLimiter` | Rate limit sul login: 5 tentativi falliti / 15min |
| `authenticate` | Verifica il JWT e aggiunge `userId` e `userRole` alla request |
| `authorize` | Factory di controllo accessi per ruolo |

## Decisioni tecniche

**Express invece di NestJS** тАФ il revisore pu├▓ leggere qualsiasi handler in pochi secondi senza conoscere il framework. NestJS non aggiunge nulla a questa scala e il suo boilerplate nasconderebbe le decisioni di design.

**Persistenza in memoria con pattern service/store** тАФ lo store vive in `src/data/store.ts` e i services sono l'unico layer che lo tocca. Passare a PostgreSQL significa sostituire quel file; il resto del codice non cambia.

**Cache Redis per la feed** тАФ la prima pagina anonima (`GET /v1/feed?limit=N`) viene cacheata su Redis con TTL di 60s. Le request autenticate vengono servite fresche per calcolare `likedByMe` per utente. A ogni like, unlike, commento o modifica, la cache si invalida con `KEYS feed:first:*` + `DEL`. In produzione si sostituirebbe `KEYS` con `SCAN` per non bloccare Redis.

**Idempotenza con `Set<string>`** тАФ i like sono memorizzati come `"userId:postId"`. `addLike` e `removeLike` verificano l'esistenza prima di mutare il contatore, garantendo che le doppie chiamate siano no-op senza logica extra.

**Cursor invece di offset** тАФ la paginazione per offset sposta i risultati se vengono inseriti post durante la navigazione. Il cursor punta all'ultimo elemento visto ed ├и stabile rispetto a inserimenti concorrenti.

**Ownership sui contenuti propri** тАФ `PATCH/DELETE /v1/posts/:id` e `PATCH/DELETE` sui commenti verificano che il richiedente sia l'autore (403 altrimenti); le rotte `/v1/admin/*` permettono la moderazione su qualsiasi contenuto, protette dal middleware `authorize('admin')`.

**JWT stateless con HS256** тАФ il payload contiene `{ sub, role, iat, exp }`. Il middleware verifica la firma crittograficamente senza toccare nessuno store. Il `role` nel token evita un lookup extra per request. In produzione: RS256 con chiave privata per firmare e pubblica per verificare in ogni servizio.

**Graceful shutdown** тАФ `SIGTERM` (Docker stop) e `SIGINT` (Ctrl+C) chiudono il server HTTP in modo pulito prima di terminare il processo, loggando ogni passo con Winston.

**Zod per validazione ed env** тАФ stesso pattern nei controller (body) e in `src/config/env.ts` (variabili d'ambiente). Se l'avvio fallisce per una variabile mal configurata, l'errore indica esattamente quale variabile e perch├й.

## Cosa manca / come lo completerei

| Elemento | Stato | Come lo completerei |
|----------|-------|---------------------|
| Database | In memoria | Sostituire `src/data/store.ts` con repository PostgreSQL (`pg` o Prisma) |
| Autenticazione reale | JWT HS256 тЬУ | RS256 + refresh token con rotazione; password hashate con bcrypt |
| Cache della feed | Redis тЬУ | `SCAN` invece di `KEYS` in produzione; cache per utente con `likedByMe` |
| Rate limiting | `express-rate-limit` тЬУ | Rate limit distribuito con store Redis per istanze multiple |
| Paginazione commenti | Cursor тЬУ (backend) | Consumare `nextCursor` anche nella UI mobile |
| Ruoli avanzati | `user` / `admin` тЬУ | RBAC granulare con permessi per risorsa |
